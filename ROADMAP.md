## 3. 스팀 게임 수준 이펙트로 올리기 위한 분석

### 3.1 현재 이펙트가 "허접해 보이는" 기술적 이유

| 현재 | 스팀 AAA 게임 | 차이 |
|------|-------------|------|
| CSS `linear-gradient` 오렌지색 | 실제 불 텍스처 (노이즈 기반) | 불이 "그림" vs "유체" |
| Canvas 2D 원형 파티클 30개 | GPU 파티클 1000~5000개 | 밀도 30배 차이 |
| `filter: sepia(1)` 로 타들어가기 | **디졸브 쉐이더** (노이즈 맵 threshold) | 진짜로 녹듯이 탐 |
| Web Audio 합성음 | 실제 녹음된 화염/충격 사운드 | "삐~" vs "쾅" |
| 이펙트만 재생 | 블룸, 모션블러, 화면 왜곡, 슬로우 모션 | 후처리 유무 |

**핵심 누락**: **WebGL 쉐이더** (GPU에서 돌아가는 그림 그리기 프로그램)가 없음.
현재 코드는 CPU로 점 찍는 수준이라 GPU가 놀고 있음.

### 3.2 "웹 = 약함"은 2015년 이전 이야기

**지금 브라우저가 가진 기술**:
- **WebGL 2.0** — GPU 직접 제어, 데스크톱 게임과 같은 쉐이더 언어 (GLSL)
- **WebGPU** (2023~) — 차세대 그래픽 API, 거의 네이티브 수준
- **Web Audio API** — 3D 위치 오디오, 리버브, 실시간 DSP
- **OffscreenCanvas + Web Workers** — 멀티스레드 렌더링
- **WebAssembly** — C++ 게임 엔진을 브라우저에서 그대로 실행 가능

### 3.3 Realm of Fates에 필요한 기술 체크리스트

| 기술 | 웹 가능? | 어떻게 | 어떤 게임이 씀 |
|------|---------|--------|-------------|
| 파티클 시스템 (수천 개) | ✅ | PixiJS / Three.js | 모든 인디 카드 게임 |
| 카드 광원/그림자 | ✅ | CSS `filter` + WebGL | Hearthstone |
| 디졸브(녹아 사라짐) 쉐이더 | ✅ | GLSL fragment shader | Slay the Spire |
| 블룸(빛 번짐) 후처리 | ✅ | PixiJS Advanced Bloom | Balatro |
| 카메라 흔들림/줌 | ✅ | CSS transform | 전부 |
| 3D 카드 회전 | ✅ | CSS `transform: rotateY` 또는 Three.js | Marvel Snap |
| 실시간 물결/왜곡 | ✅ | Displacement map 쉐이더 | Darkest Dungeon |
| 슬로우 모션 | ✅ | `requestAnimationFrame` 속도 조절 | 전부 |
| 스크린 플래시/비네팅 | ✅ | Canvas overlay | Dead Cells |
| 스프라이트 애니메이션 | ✅ | PixiJS AnimatedSprite | 2D 게임 전부 |
| 뼈대 애니메이션 (Spine) | ✅ | `pixi-spine` 플러그인 | Slay the Spire 캐릭터 |
| 버텍스 쉐이더 (3D) | ✅ | Three.js ShaderMaterial | 3D 게임 전부 |
| 리버브 3D 오디오 | ✅ | Web Audio ConvolverNode | AAA 수준 |
| 게임패드 지원 | ✅ | Gamepad API | 스팀 게임 필수 |
| 세이브 파일 | ✅ | localStorage / IndexedDB | 전부 |

**결론**: 하나도 "못함"이 없음. **웹 스택 유지로도 스팀 인디 카드 게임 수준 도달 가능.**

## 4. 기술 선택지 비교

### 4.1 전체 비교표

| 엔진/스택 | 이펙트 한계 | 학습 곡선 | 기존 코드 | 스팀 배포 | 팀 합의 | 총평 |
|----------|-----------|---------|----------|---------|-------|------|
| **현재 그대로 (HTML+JS)** | 80% | 0 (이미 앎) | 100% 유지 | ⚠️ Electron 래핑 | 🟢 쉬움 | 현실적 |
| **PixiJS + HTML** | 85% | ⭐⭐ | 80% 재활용 | ⚠️ Electron | 🟢 쉬움 | **최고 가성비** |
| **Phaser 3** | 85% | ⭐⭐⭐ | 30% 재활용 | ⚠️ Electron | 🟡 중간 | 웹게임 특화 |
| **Godot 4 (GDScript)** | 95% | ⭐⭐⭐⭐ | 0% 재활용 | ✅ 원클릭 | 🟠 어려움 | 스팀 인디 표준 |
| **Unity 2D (C#)** | 100% | ⭐⭐⭐⭐⭐ | 0% 재활용 | ✅ 원클릭 | 🔴 매우 어려움 | AAA 급 |
| **LÖVE2D (Lua)** | 90% | ⭐⭐⭐ | 0% 재활용 | ✅ 가능 | 🟠 어려움 | Balatro 루트 |
| **Defold (Lua)** | 90% | ⭐⭐⭐ | 0% 재활용 | ✅ 원클릭 | 🟠 어려움 | 모바일 강점 |

### 4.2 "언어를 바꾼다"의 진짜 의미

**팀원이 3개월간 작성한 5285줄을 전부 버려야 함.** 이미지/사운드 에셋만 재활용 가능.

| 재작성 기간 | 시나리오 |
|------------|---------|
| 3~6개월 | 1인 파트타임 |
| 1~2개월 | 2인 풀타임 |
| 영원히 현재 엔진 | 점진 개선 |

### 4.3 실제 상업 게임 사례 (웹 기술로 만든 성공작)

| 게임 | 기술 | 성과 |
|------|------|------|
| **Cookie Clicker** | 순수 HTML + JS | 수천만 플레이, 스팀 출시 |
| **Realm of the Mad God** | Flash → HTML5 | 스팀 출시, MMO |
| **Slither.io** | HTML5 | 누적 5억 다운로드 |
| **Vampire Survivors (초기)** | HTML5 + Phaser | 스팀 대성공 (후에 C++ 포팅) |
| **Balatro** | LÖVE2D + Lua | 2024 스팀 대박, 1000만 장 판매 |

**핵심**: 엔진이 성공의 열쇠가 아님. **게임성과 아트**가 핵심.

## 5. 시나리오별 권장 경로

### 🟢 시나리오 A: "지금 게임 유지 + 퀄리티 업그레이드" (권장)

**→ 현재 스택 유지 + PixiJS 추가 + 구조 리팩토링**

- 팀원 반발 거의 없음 (기술 유지)
- 기존 코드 안 버림
- PixiJS로 파티클/블러/쉐이더 접근 가능 → 이펙트 30% → 85%
- 구조 리팩토링으로 협업 문제 해결
- 스팀 출시는 **Electron 또는 Tauri**로 래핑
- Balatro 수준까진 현실적 도달 가능
- **리스크**: 낮음

### 🟡 시나리오 B: "처음부터 스팀 인디 표준" — Godot 4

- 스팀 인디 개발자 최다 사용 엔진 (Unity 정책 논란 이후 이주 증가)
- GDScript는 Python 유사 → JS 개발자 1~2주 학습
- 내장 파티클, 쉐이더 에디터, 스팀 API 원클릭
- 오픈소스 + 무료 + 로열티 0%
- **리스크**: 중간 (팀원 포기 시 프로젝트 사망)

### 🔴 시나리오 C: "AAA 게임 수준" — Unity 2D

- Hearthstone, Legends of Runeterra가 쓰는 엔진
- 에셋 스토어에 카드 게임 템플릿 다수
- C# 배워야 함 (3~6개월)
- **리스크**: 매우 높음 (학습 장벽이 팀 무너뜨릴 수 있음)

## 6. 최종 권장 로드맵 (시나리오 A 기반)

### Phase 0: 이펙트 브랜치 마무리 (1~2일)
- 현재 `feature/defeat-animation` 브랜치 정리
- 팀원에게 보여주고 반응 확인
- 중간 결과: **패배 연출 데모 완성**

### Phase 1: 구조 리팩토링 (1~2주) ⭐ **최우선**
- `index.html` → 10~15개 파일로 분리
- **빌드 도구 없이** 순수 `<script>` 태그 분리
- GitHub Pages 배포 구조 유지
- 중간 결과: **협업 편해짐, 충돌 0**

### Phase 2: PixiJS 도입 + 이펙트 고도화 (2~4주)
- 파티클 시스템 PixiJS로 교체
- 블룸, 블러, 쉐이더 필터 도입
- 디졸브 쉐이더 작성 (카드 소각)
- 포스트 프로세싱 추가
- 중간 결과: **이펙트가 게임 같아짐**

### Phase 3: Electron/Tauri 래핑 + 스팀 배포 준비 (1~2주)
- 웹 게임 그대로 데스크탑 앱으로 포장
- 스팀 SDK 연동
- 게임패드 지원 추가
- 중간 결과: **스팀 출시 가능 상태**

### (선택) Phase 4: Godot 재작성
- Phase 3까지 해보고도 한계 느낄 때만
- 현실적으로 필요 없을 가능성 높음

**누적 기간**: 2~3개월 (Phase 0~3까지)

## 7. 구조 리팩토링 상세 계획

### 7.1 제안 파일 구조

```
realm-of-fates/
├── index.html               ← HTML 뼈대만 (~400줄)
├── css/
│   ├── base.css             ← 공통 스타일, 버튼
│   ├── screens.css          ← 각 화면 스타일
│   ├── battle.css           ← 전투 스타일
│   └── effects.css          ← 애니메이션/이펙트 @keyframes
├── js/
│   ├── data.js              ← UNITS, SKILLS, RELICS 상수
│   ├── auth.js              ← 로그인/회원가입
│   ├── sfx.js               ← 사운드 시스템
│   ├── ui.js                ← 공통 UI, 모달
│   ├── town.js              ← 마을/건물/업그레이드
│   ├── tavern.js            ← 술집/영입
│   ├── battle.js            ← 전투 로직
│   ├── cards.js             ← 카드/덱/장비
│   ├── fx.js                ← 파티클, 이펙트
│   └── game.js              ← 메인 Game 객체, 진입점
├── img/                     ← 그대로 유지
└── snd/                     ← 그대로 유지
```

**목표**: 각 파일 500줄 이하

### 7.2 리팩토링 원칙

1. **빌드 도구 도입 금지** — Webpack/Vite 쓰지 말 것
   - 이유: GitHub Pages 배포 단순성 유지, 팀원 진입장벽 0
2. **팀원 사전 합의 필수**
   - 이유: 합의 없이 머지하면 팀원 로컬 브랜치 충돌 지옥
3. **제안서 먼저 작성 (REFACTOR_PLAN.md)**
   - 이유: 코드 건드리기 전 방향성 리뷰받기
4. **한 번에 다 바꾸지 말기**
   - 이유: 조각조각 롤아웃해야 롤백 가능

### 7.3 왜 리팩토링이 이펙트보다 먼저인가

| 관점 | 이펙트 먼저 | 구조 먼저 |
|------|----------|----------|
| 팀원 충돌 | 🔴 매번 충돌 | 🟢 파일 분리 후 각자 영역 |
| 검색/수정 | 🔴 5000줄 스크롤 | 🟢 해당 파일 바로 열기 |
| 새 기능 | 🔴 어디 넣을지 고민 | 🟢 명확한 위치 |
| 에디터 | 🟡 VSCode 버벅임 | 🟢 쾌적 |
| 신규 협업자 | 🔴 5000줄 보고 도망 | 🟢 파일별 파악 |

**리팩토링은 어떤 엔진을 선택하든 무조건 해야 하는 작업.**

## 10. 핵심 요약 (TL;DR)

1. **현재 허접한 이유**: 기술 한계 아님. 라이브러리 미사용 + 쉐이더 미사용 + 에셋 퀄리티
2. **웹 스택으로 스팀 게임 수준 가능**: Balatro, Cookie Clicker, Slither.io가 증명
3. **엔진 바꾸지 말 것**: 기존 3개월치 코드 버리고 3~6개월 재작성 필요
4. **1순위 작업**: 구조 리팩토링 (어떤 선택이든 먼저 필요)
5. **로드맵**: 리팩토링 → PixiJS → 쉐이더 → Electron → 스팀
6. **리팩토링 원칙**: 팀원 합의 필수, 빌드 도구 금지, 제안서 먼저
7. **에셋 업그레이드도 병행**: 기술만큼 중요
